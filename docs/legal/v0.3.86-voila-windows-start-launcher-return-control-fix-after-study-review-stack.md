# v0.3.86 — Voila Windows START launcher return-control fix after Study/Review stack

## Result

PASS local.

## Summary

This milestone fixes a Windows package START launcher return-control regression found during the v0.3.85 ZIP START/STOP smoke attempt after the Study/Review/Course Tools stack.

## Regression found

The rebuilt v0.3.84 ZIP candidate was valid as a package candidate, but START/STOP smoke exposed a launcher issue:

- ZIP SHA256 verification: PASS.
- ZIP extraction: PASS.
- Launcher presence: PASS.
- Ports free before START: PASS.
- API could start and respond during diagnostics.
- START launcher did not return control reliably.
- In one smoke attempt, START returned only after a long delay and the API was no longer reachable.

Therefore the ZIP candidate START/STOP smoke was not marked PASS.

## Root cause

The generated START launcher process model was not reliably detached from the PowerShell/cmd wrapper on Windows.

Intermediate attempts were tested and rejected:

- Redirecting stdout/stderr while using Start-Process was not sufficient.
- A cmd.exe /c start runner still caused frozen/bounded-smoke failures.

## Implemented fix

Updated `scripts/release/create-complete-windows-runtime-source.ps1` so generated Windows runtime START launchers:

- Generate `runtime/state/run-voila-api.cmd`.
- Launch the API through `Invoke-CimMethod Win32_Process.Create`.
- Use `cmd.exe /d /c` to run the generated API runner.
- Resolve the real API owner PID from port 8787.
- Write the resolved PID to `runtime/state/voila-api.pid`.
- Write logs to:
  - `runtime/logs/start-voila.log`
  - `runtime/logs/voila-api.out.log`
  - `runtime/logs/voila-api.err.log`
- Preserve bounded startup verification.
- Preserve package-owned STOP behavior.
