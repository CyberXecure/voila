# Voila! UI Remaining Labels Plan

Milestone: v0.2.58-public-beta-language-pack-ui-remaining-labels-plan
Status: planning
Scope: documentation only; no UI code changes, no runtime changes, no schema changes, no GitHub release upload, no tag, no public ZIP publish

## Goal

This milestone documents a remaining-label strategy after the controlled UI language-pack batches completed in v0.2.50, v0.2.53, and v0.2.56.

The goal is to avoid forcing existing `ui.*` keys onto mismatched long helper text, generated output, log text, CSS matches, or developer diagnostics.

## Baseline

Completed UI language-pack work:

- v0.2.50 integrated `ui.upload_pdf`, `ui.generate_course`, and `ui.open_course`
- v0.2.53 integrated `ui.figures`, `ui.study`, `ui.progress`, and `ui.delete_from_library`
- v0.2.56 integrated `ui.edit_crops`, `ui.review_weak`, `ui.generated`, `ui.source_mode`, and an extra `ui.progress` fallback
- v0.2.57 documented the v0.2.56 UI next-batch integration

## Scan result

The remaining-label scan is exploratory.

Important finding:

```text
Generic terms such as Back are noisy because they also match CSS words such as background.
```

Therefore, future scans should use precise selectors and inspect context before patching.

## Remaining label categories

### 1. Short UI labels

These are possible future candidates for existing key reuse or new `ui.*` keys.

Examples:

```text
Back
Tools
Course
Lessons
Library
Logs
Open course
Study mode
Course Tools
Quick Tools
```

Recommended handling:

```text
inspect exact context first
reuse existing keys only when semantically exact
add new keys only in a dedicated planning milestone
```

### 2. Existing translated legacy keys

Some remaining strings already have legacy non-`ui.*` keys.

Examples:

```text
back
study
progress
review
reset
back_to_lessons
study_coverage
reset_study_progress
```

Recommended handling:

```text
keep existing fallback behavior
do not replace blindly
migrate to ui.* only when visible UI usage is clear and smoke-tested
```

### 3. Long helper text

These should not be forced into short `ui.*` labels.

Examples:

```text
Choose a PDF from your computer. It will be saved locally in data/input.
No PDF files found. Put a PDF in data/input, then refresh this page.
Read the generated course with navigation.
Choose a lesson, read it, then study only that lesson.
Practice questions generated from the course.
Correct OCR text page by page.
Manually edit figure crops.
Return to the main course library.
```

Recommended handling:

```text
create dedicated message/helper keys later
do not map to short labels like ui.choose_file
```

### 4. Generated content / document content

These should normally stay untouched because they are part of generated course files, logs, OCR output, or document content.

Examples:

```text
Generated by Voila!
Voila! course files generated successfully.
Questions generated
Lessons generated
```

Recommended handling:

```text
do not localize in UI label batches
classify separately as generated output/log text
```

### 5. Debug / error / developer text

These should not be part of simple UI label localization.

Examples:

```text
traceback output
fallback diagnostics
console warnings
developer-only route messages
```

Recommended handling:

```text
leave unchanged unless a later diagnostics-localization milestone is explicitly planned
```

## Recommended next scan method

Future scans should avoid broad patterns that cause false positives.

Better strategy:

```powershell
Select-String -Path .\services\api\web_app.py -Pattern '<a |<button|<h1|<h2|<title>|class="notice"|class="meta"|card\('
```

Then manually classify the visible text.

Avoid scanning broad words such as:

```text
Back
Course
Generated
```

without context filters.
